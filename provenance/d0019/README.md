# Ontogen contract convergence — vault-status projection, domain-op actions, and the single-source-of-truth decision

## Summary

The workspace is converging on **one dashboard over a pluggable transport**: a shared Nuxt
layer (`packages/dashboard`) whose pages speak the ontogen `Transport` contract, so the same UI
runs over HTTP (the `apps/web` Bun daemon) and over Tauri IPC (`apps/desktop`, Rust backend).
The backend convergence step (PR #250) reshaped the Bun daemon's `/api/vaults` CRUD onto the
identity-only ontogen `Vault`, which **broke the existing `apps/web` editor at runtime** because
that editor renders *derived status* (pass/fail + findings + drift) the ontogen `Vault` doesn't
carry. PR #251 restored it with a **stopgap** read route (`GET /api/vault-status`) that the Bun
daemon computes by hand.

The editor cannot yet run purely over the ontogen contract for **two** reasons: the contract has
**no status read model** (status is normalized into separate `ScanRun` / `FindingRecord`
resources, with no "vault-with-latest-status" projection and no live-status channel), and it
models **none of the editor's domain operations** (`validate`, `check`/drift, `init`,
`config` + `config/files` editing, `watch`). This record establishes that both gaps are
**closeable within ontogen** — entities extend by annotated Rust structs and non-CRUD operations
extend by `#[ontogen] pub fn`s that auto-generate IPC + HTTP handlers and TS bindings — and lays
out the plan to close them.

The plan is three workstreams: **(A)** a first-class `vault-status` projection (retiring the
#251 stopgap), **(B)** the domain operations as ontogen actions, and **(C)** the decision that
gates the *cost* of A and B: whether the web surface keeps a **hand-mirrored Bun daemon**
(author every addition twice) or **converges onto the already-generated Rust HTTP server**
(author once, retire the Bun daemon). **Recommendation: pursue C-option-2 (converge on the Rust
HTTP server), de-risked first** — mirroring how the wasm-engine packaging question was handled
(the `spike/engine-wasm` de-risk, PR #249): prove parity before committing. A/B are cheap once C
is settled and expensive if it isn't.

^summary

## Context

Three surfaces share one UI and (aspirationally) one API contract:

1. **`apps/desktop`** ([[D-0018-desktop-tauri-nuxt]]) — a Tauri v2 shell over a Nuxt SPA, backed
   by a Rust service layer **generated by ontogen** (`github.com/sksizer/rust-ontogen`) from
   annotated entities in `apps/desktop/src-tauri/src/schema/`. Transport is Tauri IPC.
2. **`apps/web`** ([[D-0012-distribution-single-exec-and-web-ui]]) — the single-binary + loopback
   **Bun daemon** serving a Nuxt SPA + JSON API over `packages/core`'s runner. Transport is HTTP
   (+ SSE for live status). The daemon's `/api/*` handlers are **hand-written TS**
   (`apps/web/src/daemon/api.ts`) that *mirror* the ontogen contract shape.
3. **Latent surface ③** — a standalone Rust web server. Its HTTP transport is *already generated*
   by the ontogen pipeline (`apps/desktop/src-tauri/src/api/transport/http/generated.rs`); it is
   simply not the primary web server today.

The shared dashboard (`packages/dashboard`) consumes the ontogen `Transport` interface; its
`bindings/` are a **web-safe slice** of the desktop's generated bindings, produced by
`packages/dashboard/scripts/sync-bindings.ts` (strips the Tauri-IPC transport + `@tauri-apps`
imports so the web bundle carries no Tauri SDK).

**The trigger.** PR #250 converged the Bun daemon's base `/api/vaults` (+ `:id`) routes onto the
flat ontogen `Vault` (`apps/web/types/ontogen.ts`). CI stayed green because nothing exercises the
UI against the daemon, but the editor broke at runtime: `useApi.listVaults()`/`getVault()`
expected the status-bearing `VaultStatus` (`apps/web/types/api.ts`). PR #251 restored it with a
hand-computed `GET /api/vault-status` read model (a join of the registry + the in-memory
`StatusStore`). That is a **patch on one surface**, not contract convergence — which is what this
record is about.

## The gap — why the editor can't run on the ontogen contract today

### Gap A — the ontogen contract has no "vault status"

The ontogen `Vault` is **identity only** (`id`, `name`, `root`/`path`, `config_path`,
`watch_enabled`, `schedule`, timestamps — `apps/desktop/src-tauri/src/schema/vault.rs`). The
editor's `VaultStatus` (`apps/web/types/api.ts`) adds the **derived** `state`
(`green` / `findings` / `drift` / `running` / `error`), the latest `RunResult` (findings), the
latest `DriftResult`, and any `error`. Ontogen normalizes this apart into `ScanRun` and
`FindingRecord` resources. To render the dashboard over the contract as it stands, a client would
have to:

- fetch **all** `scan-runs` + **all** `finding-records` and **join client-side** — there is no
  "vault with its latest status" projection and no "latest scan-run per vault" query;
- accept blank status until a scan runs — the **Bun daemon's** `ScanRun`/`FindingRecord` stores are
  **in-memory** (the desktop's are already SQLite-persisted; the web side's layer-3 SQLite is
  deferred per [[D-0012-distribution-single-exec-and-web-ui]]);
- separately consume **live status**, which today is SSE (`GET /api/events`) and is **not part of
  the `Transport` contract** at all.

### Gap B — the ontogen contract models none of the editor's domain operations

The editor is far more than CRUD. None of these exist in the ontogen contract (which is generic
entity CRUD + the `scans/now`, `echo`, `openers/*` actions):

| Operation | Route today (Bun, kept as legacy) | Purpose |
| --- | --- | --- |
| `validate` | `POST /api/vaults/:id/validate` | run the corpus → findings |
| `check` | `GET /api/vaults/:id/check` | drift (committed contract vs. re-inferred) |
| `init` | `POST /api/vaults/:id/init` | scaffold contract files |
| `config` | `GET`/`PUT /api/vaults/:id/config` | read/replace the router config YAML |
| `config/files` | `GET`/`PUT /api/vaults/:id/config/files` | the **contract-authoring** editor (router + referenced `*.contract.yaml`) |
| `watch` | `POST /api/vaults/:id/watch` | toggle file-watching |
| live status | `GET /api/events` (SSE) | push state transitions |

Roughly **half of what the editor does has no ontogen representation.** Config-file editing is the
richest — it is the whole contract-authoring surface.

## How ontogen extends (the enabling facts)

The ontogen pipeline runs from `apps/desktop/src-tauri/build.rs` on every `cargo build` over
`src/`. There are **two** extension points, and both flow all the way to the generated TS bindings:

1. **Entities** — annotated structs in `src/schema/*.rs`
   (`#[derive(OntologyEntity)]`, `#[ontology(entity, table = "…")]`, `#[ontology(id)]`).
   → SeaORM/SQLite + CRUD store (+ hook stubs in `src/store/hooks/`) + `Create*/Update*` DTOs +
   IPC & HTTP transports + TS bindings.
2. **Custom actions** — ontogen **scans `src/api/v1/` for plain `#[ontogen] pub fn`s** and
   generates their per-transport handlers (IPC + HTTP) *and* TS transport methods automatically.
   Existing templates: `src/api/v1/scan.rs` (`scan_now(state, vault_id) -> Result<ScanRun>`
   → `POST /api/scans/now`), `src/api/v1/echo.rs`, `src/api/v1/openers.rs`. Return types may be
   **hand-owned DTOs** (e.g. `OpenerInfo`, `OpenPreview` in `src/schema/openers.rs`) that are not
   entities but still cross the generated TS surface.

**Conclusion: both gaps are closeable within ontogen.** Gap A closes with a `vault_status` action
returning a hand-owned `VaultStatus` DTO (a join projection). Gap B closes with one
`#[ontogen] fn` per operation. This is the fact that could not be confirmed when #251 shipped the
stopgap; it now is.

**The catch — two implementations of one contract.** The above generates the **desktop** stack
(Rust → IPC + Rust HTTP + bindings). The **`apps/web` Bun daemon is a separate hand-written TS
re-implementation** of the same shape over `packages/core`. So *today* every contract addition must
be authored twice: once as Rust schema/`#[ontogen] fn` (generated), once by hand in
`apps/web/src/daemon`. That duplication is the real cost driver, and it is the subject of
workstream C.

## Plan

### Workstream A — status as a first-class read (retires the #251 stopgap)

- Add a `vault_status` ontogen action returning a hand-owned `VaultStatus` DTO: the join of a
  `Vault` with its latest `ScanRun` + findings summary. It generates as a transport method on both
  surfaces, so the **shared dashboard** (not just the bespoke editor) can read status.
- `ScanRun` / `FindingRecord` are already SQLite-persisted on desktop; reconcile the Bun daemon's
  in-memory stores as part of C.
- **Open question (confirm early):** does ontogen model **streaming / subscriptions**? That decides
  whether live status becomes a `Transport` channel or stays an SSE adjunct.

### Workstream B — domain operations as ontogen actions

- One `#[ontogen] pub fn` per operation in `src/api/v1/`, with small hand-owned DTOs:
  read ops (`validate`, `check`) first → write ops (`config`, `config/files`) → `watch`.
- Config-file editing likely wants its own DTO cluster (router + referenced contract files, parse
  verdicts).
- Once these land in the bindings, the dashboard layer can call them and the editor can migrate off
  its bespoke `useApi`.

### Workstream C — single source of truth (the decision that gates A and B)

- **Option 1 — keep both, mirror by hand.** Every A/B addition authored in Rust *and* hand-written
  in the Bun daemon. Simple to reason about; **doubles the work per feature** and re-opens drift
  (exactly the drift #250 → #251 exposed).
- **Option 2 — converge the web surface onto the generated Rust HTTP server** (latent surface ③).
  The Rust HTTP transport is *already generated*; make it the web server, retire the Bun daemon.
  Then A/B are authored **once** in Rust and both desktop and web inherit them. Bigger up-front
  move; it is the actual "one contract, one source" end state and removes the duplication for good.

## Decision & recommendation

**Recommended: Option 2 (converge on the Rust HTTP server), de-risked before committing.** The
duplication in Option 1 is precisely the failure mode that produced this record; carrying it
forward taxes every future contract change. Option 2 is the end state the three-surface design was
always aiming at.

But Option 2 has a real unknown — **can the generated Rust HTTP server serve what the Bun daemon
does today?** The Bun daemon's value is its `packages/core` integration (config discovery mirroring
the CLI, corpus runs, drift inference). The de-risk (this record's immediate next step) checks
whether the Rust engine layer (`apps/desktop/src-tauri/src/engine/`) + generated HTTP transport
reaches parity with the Bun daemon's `packages/core` path — same corpus results, config resolution,
finding shapes — for a fixture vault. This mirrors how the wasm-engine question was handled: prove
the foundation, then collapse onto it.

If the de-risk says **go**, A and B are authored once in Rust. If it says **not yet**, fall back to
Option 1 for A and the read-op half of B (cheap, high-value), and keep Option 2 as the tracked
target.

## Workstream-C de-risk findings (2026-07-14)

A static investigation of whether the generated Rust HTTP server can replace the Bun daemon.
**Verdict: GO on Option 2.**

1. **The full ontogen contract is already generated as HTTP routes.** `entity_routes()`
   (`apps/desktop/src-tauri/src/api/transport/http/generated.rs`) mounts vaults / scan-runs /
   finding-records / opener-preferences CRUD **plus** the `scans/now`, `echo`, and `openers/*`
   actions — i.e. exactly the Bun daemon's ontogen surface, generated rather than hand-written.
2. **A loopback HTTP host already exists.** `apps/desktop/src-tauri/src/daemon.rs` (`serve()`)
   mounts those routes on an Axum `TcpListener` "the way the Bun daemon serves its JSON API" — but
   it sits behind the **`daemon` Cargo feature (off by default)** and is **wired to nothing**:
   `main.rs` only launches Tauri, and `daemon::serve` has no caller. Standing it up = enable the
   feature + add an entry point (a CLI subcommand or a second bin) that constructs `AppState`
   (DB, engine, registry) outside Tauri and calls `serve`. Bounded, modest work — the routes,
   the hard part, are generated.
3. **Engine/findings parity is designed-in, not hoped-for.** The Rust engine
   (`src/engine/native.rs`) wraps `crates/markdown-contract-engine` (a path dep) — the same engine
   pinned to the shared corpus goldens with `packages/core`. `scan_now` runs it and records a
   `ScanRun` + `FindingRecord`s. Finding parity is a tested invariant.
4. **The domain-op cost is the same either way (and cheaper here).** The editor's ops
   (`validate`/`check`/`init`/`config`/`config-files`/`watch` — Gap B) exist on **neither** surface
   as ontogen operations today. Converging on Rust means authoring them **once** as
   `#[ontogen] fn`s (Option 1 authors them twice).
5. **Divergences to accept + document:**
   - **JS/MJS programmatic configs.** The Bun daemon's `resolveConfig` (`apps/web/src/daemon/routes.ts`)
     dynamic-imports `.js`/`.mjs` config modules; a Rust server can't. YAML-only in Rust —
     acceptable for the dashboard (vaults use `markdown-contract.yaml`), but a real CLI power-feature
     that would not carry over.
   - **Single-binary packaging** shifts from `bun build --compile` to a Rust `cargo build
     --features daemon` + serve entry. Arguably more robust (the bun-compile path hit environment
     signing/exec issues), but a distribution change to plan for.

**Empirical check:** `cargo check -p markdown-contract-desktop --features daemon` → **clean
(exit 0)**. The feature-gated `serve()` + generated Axum router + `AppState` compile today; the
serve path is buildable code, not just latent scaffolding — only an entry point that calls it is
missing.

**Net:** the plumbing (complete generated router + existing `serve()`), the parity (golden-tested
engine), and the extension mechanism (`#[ontogen]` fns) all exist. Option 2's remaining work is a
modest serve entry point plus authoring Gap-B once in Rust — strictly less than Option 1's
double-authoring. **Recommendation confirmed.**

## Sequencing

1. **De-risk C** — Rust-HTTP-vs-Bun-daemon parity for a fixture vault. *(immediate next step)*
2. **A** — the `vault_status` projection (retires the #251 stopgap).
3. **B**, incrementally — `validate` / `check` → `config` / `config-files` → `watch`.
4. **Migrate the editor** onto the transport, deleting the bespoke `apps/web/ui/app/composables/useApi.ts`.

## Open questions

- Does ontogen model **streaming/subscriptions** for live status, or does SSE stay an adjunct?
- Does the generated Rust HTTP server reach **`packages/core` parity** (config discovery, corpus
  runs, drift, finding shapes)? — the workstream-C de-risk.
- Web-side **persistence**: adopt the desktop's SQLite via the Rust server (Option 2), or add a
  layer-3 store to the Bun daemon (Option 1)?
- Config-file editing DTO shape — one `config_file` resource, or an action returning a file list?

## Status quo (the #251 stopgap this record supersedes)

PR #251 added a read-only `GET /api/vault-status` (+ `/:id`) to the **Bun daemon only**
(`apps/web/src/daemon/api.ts`) — a hand-computed join of the registry + `StatusStore`. It
un-broke the editor without touching #250's ontogen CRUD, and it is explicitly interim: workstream
A replaces it with a generated, cross-surface `vault_status` projection.
